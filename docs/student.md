
这份技术文档是为 ** 负责人3（学生能力建模与大模型工程负责人）**设计的。该模块的核心目标是将简历（PDF/Word）及用户录入的非结构化文本，通过 LLM 转化为系统可计算的结构化能力向量 。

----------

## 学生能力建模模块（Student Profiling Module）技术文档

### 1. 模块目标

-   **非结构化提取**：利用大模型从简历文本中精准抽取技能、项目、经历等关键信息 。
    
-   **能力标准化**：对抽取的原始标签进行结构化处理（如：将“熟练使用 SpringBoot”映射为技能字段并保留原文证据） 。
    
-   **向量化输出**：生成符合负责人 1 定义的标准化能力向量结构，供匹配引擎调用 。
    

### 2. 核心数据流程

1.  **文本预处理**：使用 `PyMuPDF` 或 `python-docx` 提取简历内容，并进行清洗，保留关键语义结构。
    
2.  **LLM 抽取（LangChain）**：采用 `Structured Output` 模式，基于 Pydantic 定义的 Schema 强制模型输出固定格式的 JSON 。
    
3.  **能力评分与量化**：基于简历体现的综合素质，由大语言模型对职业素养和发展潜力进行定性评估（高/中/低）。
    

----------

|--|--|--|--|--|--|
|说明|提取学生技能集合，包含名称与证据|提取学生相关证书|提取项目经历与实习经历信息|根据简历的信息判断职业素养评分|根据简历信息判断发展潜力评分|

### 4. 技术实现方案

#### 4.1 简历解析 Prompt 策略

为确保输出稳定性，提示词需明确约束评分选项和 JSON 结构：

```text
你是专业的简历信息提取分析师，需要从以下简历文本中精准提取指定信息，并严格按照要求的 JSON 格式输出。
评分必须仅限“高”、“中”、“低”三个选项，evidence 字段需准确引用简历原文中的对应表述。

【提取字段要求】
1. skills：数组类型，每个元素包含 name（技能名称）、evidence（原文证据）；
2. certificates：数组类型，每个元素包含 name（证书名称）、evidence（原文证据）；
3. Experience：数组类型，每个元素包含 name（项目/实习经历名称）、evidence（原文证据）；
4. Professionalism：对象类型，包含 level（评分，可选值：高、中、低），基于责任心、执行力评估；
5. Potential：对象类型，包含 level（评分，可选值：高、中、低），基于学习能力、成长空间评估。

【输出要求】
- 严格遵循 JSON 格式；
- 无对应信息的字段填充空数组，评分字段默认填“低”；
- evidence 字段必须是简历原文的精准截取。
```
